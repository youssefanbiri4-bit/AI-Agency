# Incident Response Playbook

**دليل:** الاستجابة للحوادث  
**المسؤول:** Security Agent + DevOps Agent  
**المستوى:** P0-P1 (حرج)

---

## 1. التعريف

هذا الدليل يصف الخطوات الواجب اتباعها عند اكتشاف حادثة أمنية أو إنتاجية في البيئة الحية.

## 2. أنواع الحوادث

| النوع | الوصف | المدة المستهدفة |
|-------|-------|----------------|
| **P0 — Critical** | تعطل كامل، فقدان بيانات، ثغرة أمنية | < 15 دقيقة |
| **P1 — High** | تعطل ميزة رئيسية، تدهور أداء | < 30 دقيقة |
| **P2 — Medium** | تعطل ميزة ثانوية | < 4 ساعات |
| **P3 — Low** | مشكلة تجميلية | < 1 أسبوع |

## 3. أدوات التحقيق

| الأداة | الاستخدام | الرابط |
|--------|-----------|--------|
| Vercel Logs | سجلات الخادم والـ API | Vercel Dashboard |
| Sentry | تقارير الأخطاء | Sentry Dashboard |
| Supabase Logs | استعلامات قاعدة البيانات | Supabase Dashboard |
| Health Check | فحص الخدمات | /api/health |

## 4. إجراءات الاستجابة

### الخطوة 1: الاكتشاف
- أي وكيل يمكنه الإبلاغ عن حادثة
- إشعار فوري عبر قناة الطوارئ

### الخطوة 2: التقييم
```
Is it a security incident?
  → YES: تعطيل الميزة فوراً + إشعار CTO
  → NO: هل هو فشل إنتاجي؟
    → YES: rollback فوري
    → NO: فتح bug report
```

### الخطوة 3: الاحتواء
- **Security Incident:** تعطيل الميزة المصابة
- **Production Failure:** rollback إلى آخر إصدار مستقر
- **Data Issue:** freeze على الجدول المصاب

### الخطوة 4: الإصلاح
- Hotfix مع Expedited Review
- Security Agent + Architect Agent فقط
- CTO Approval فورية

### الخطوة 5: الاستعادة
- نشر hotfix
- التحقق من الإصلاح
- إعادة تمكين الميزة

### الخطوة 6: المتابعة
- Post-Mortem خلال 24 ساعة
- توثيق الدروس المستفادة
- تحديث الإجراءات الوقائية

```
Action Taken:
[الإجراءات المتخذة]

Next Steps:
[الخطوات التالية]

```

## 6. قناة الاتصال

- جميع الاتصالات أثناء الحادثة عبر قناة الطوارئ المخصصة
- تحديثات كل 15 دقيقة للحوادث P0/P1
- تحديثات كل ساعة للحوادث P2
- لا تغيير في الحالة دون إشعار الفريق
